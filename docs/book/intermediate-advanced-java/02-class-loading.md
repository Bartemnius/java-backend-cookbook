# Class Loading

## Simple Explanation

Class loading is the process of bringing a `.class` file into the JVM so the program can use it.

Example:

```java
User user = new User();
```

Before the JVM can create a `User` object, it needs the `User` class definition.

That class definition is loaded by a class loader.

Simple mental model:

```text
.class file -> class loader -> JVM memory -> class can be used
```

## Why It Exists

Java applications are made of many classes.

The JVM does not need to load every possible class immediately.

Classes can be loaded when needed.

This helps with:

- startup behavior,
- modularity,
- dynamic loading,
- frameworks,
- application servers,
- plugins,
- isolation between libraries.

Spring, Hibernate, application servers, build tools, and test frameworks all rely heavily on class loading behavior.

## Main Phases

Class loading is often described in phases:

```text
Loading
Linking
Initialization
```

Linking includes:

```text
Verification
Preparation
Resolution
```

You do not need to memorize every detail for most interviews.

But you should understand the high-level flow.

## Loading

Loading means the JVM finds the class bytecode and creates a class representation inside the JVM.

Sources can include:

- `.class` files,
- JAR files,
- modules,
- generated bytecode,
- custom class loader sources.

## Verification

The JVM verifies bytecode before running it.

This checks that the bytecode follows JVM rules and is safe enough to execute.

For example, bytecode should not corrupt the stack or break type safety.

## Preparation

The JVM allocates memory for static fields and sets default values.

Example:

```java
public class AppConfig {
    static int maxRetries = 3;
}
```

During preparation, `maxRetries` gets a default value first:

```text
0
```

The explicit value `3` is assigned later during initialization.

## Initialization

Initialization runs static initializers and assigns explicit static field values.

Example:

```java
public class AppConfig {
    static int maxRetries = 3;

    static {
        System.out.println("AppConfig initialized");
    }
}
```

The static block runs during class initialization.

## Parent Delegation Model

Java class loaders usually follow parent delegation.

Simple idea:

```text
Ask parent class loader first.
If parent cannot load it, try loading it yourself.
```

This helps protect core Java classes.

For example, your application should not replace `java.lang.String` with its own fake version.

## Common Interview Questions

### What is class loading?

Good answer:

```text
Class loading is the process of loading Java bytecode into the JVM so a class can
be used. It includes loading, linking, verification, preparation, and initialization.
```

### When is a class loaded?

Usually when it is first actively used.

Examples:

- creating an instance,
- accessing a static field,
- calling a static method,
- using reflection.

Exact behavior can depend on the situation, but "when first needed" is a useful interview-level answer.

### What is parent delegation?

Good answer:

```text
Parent delegation means a class loader asks its parent to load a class first.
Only if the parent cannot load it does the child class loader try. This helps
protect core Java classes and avoid class loading conflicts.
```

### What is the difference between loading and initialization?

Loading brings the class into the JVM.

Initialization runs static initializers and assigns explicit static field values.

## Common Mistakes

Mistake 1: Thinking all classes load at application startup.

Many classes are loaded lazily when first used.

Mistake 2: Confusing object creation with class loading.

A class is loaded once per class loader.

Objects can be created many times from that class.

Mistake 3: Ignoring class loader problems in frameworks.

Class loader issues can appear in application servers, plugins, tests, and complex dependency setups.

## Mid/Senior Notes

Most normal backend code does not manually create class loaders.

But class loading matters when debugging:

- `ClassNotFoundException`,
- `NoClassDefFoundError`,
- dependency conflicts,
- application server isolation,
- shading/relocation issues,
- plugin systems,
- reflection-heavy frameworks.

Example difference:

```text
ClassNotFoundException -> class could not be found when explicitly loading it
NoClassDefFoundError   -> class existed at compile time but is missing at runtime
```

Senior-level answer:

```text
I usually do not write custom class loaders, but I need to understand class loading
because frameworks, dependency conflicts, and runtime errors often involve classes
not being found, being loaded by the wrong loader, or being initialized at unexpected
times.
```

## Quick Summary

Class loading brings `.class` bytecode into the JVM.

Important phases are loading, linking, and initialization.

Verification checks bytecode safety.

Initialization runs static initializers.

Parent delegation asks parent class loaders first.

Class loading matters in framework and dependency problems.
